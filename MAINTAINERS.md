## Open ownership slots

Any of these can be co-owned with the project lead, or wholly owned by a
trusted contributor over time.

| Subsystem | Repos | What "owning it" looks like |
|-----------|-------|------------------------------|
| **Widget** | `quelora-widget-community`, `quelora-widget-enterprise`, `quelora-wasm` | Triage widget issues, review widget PRs, drive widget releases. |
| **Public API** | `quelora-public-api`, `quelora-common` | Triage API issues, review PRs, watch for breaking changes. |
| **Dashboard** | `quelora-dashboard`, `quelora-dashboard-api` | Triage admin-side issues, review UI/UX PRs. |
| **Background workers** | `quelora-worker`, `quelora-jobs` | Triage queue + job issues, performance. |
| **WordPress plugin** | `quelora-wp-plugin` | WordPress.org plugin updates, support requests. |
| **Optional modules** | `quelora-enterprise` | Surveys / gamification / ads / chat / SSE / live / P2P features. |
| **Self-hosting** | `quelora-dx-env`, this meta-repo | Docker / installer / docs ergonomics. |
| **ML moderation** | `quelora-detoxify` | Detoxify wrapper, model updates. |
| **Demo** | `quelora-demo-api` | Keep the live demo healthy. |

## How to become a co-maintainer

1. Land 2-3 meaningful PRs in a given subsystem (small refactors, bug fixes,
   doc improvements, tests — anything that shows you can work in the area).
2. Open a discussion saying you'd like to maintain that area.
3. Get push access on the relevant repo(s) and your name added here.

There's no minimum hour commitment. "I can look at issues in this area once a
week" is enough to start.

## Responsibilities

- Triage incoming issues and PRs in your area.
- Apply labels (`good first issue`, `help wanted`, `bug`, etc.).
- Review PRs to your area within ~1 week (be honest about your availability).
- Cut releases when needed (no fixed cadence — release when it's worth it).
- Be a good citizen of the [Code of Conduct](./CODE_OF_CONDUCT.md).

## Stepping down

Anyone can step down at any time — open a PR removing yourself, or just say
so in a discussion. The project would rather know than guess.
